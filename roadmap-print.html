Without these, a user signs up, posts once, and never comes back. Each can be cut into a 1–3 day PR.

Profile pages

2 days

One screen per user: name, badges, helpful points, comments they've written (anonymous posts hidden by design).

Why now. The whole "earn reputation through good replies" idea is invisible without somewhere to display it. Currently profiles don't exist as a route.
needs · user_stats table · /users/:id route

Helpful marks

2 days

Heart icon on each comment + each post. Polymorphic — same UI for both. Count is the only ranking signal we expose.

Why now. No upvote / like exists today. Without it there's no way to surface good advice or reward commenters — the whole feedback layer of the product is missing.
needs · helpful_marks table · UQ on (user, target) so one user can't spam

Stable anon handles

1 day

Each post gets a memorable handle like anon_a91f. Same across replies in that thread, different across threads.

Why now. "Posted anonymously · Posted anonymously · Posted anonymously" reads flat. Handles make multi-message threads readable. Also: move user_id off the posts table — currently identity sits next to content in the DB, which is a privacy footgun.
needs · anon_handles table · migration to drop posts.user_id

Notifications

3 days

In-app bell + email digest. Two events at launch: "someone replied to your anonymous post" and "your reply earned a helpful mark."

Why now. Posters can't see their own anonymous posts in a feed (no way to find them again). Notifications are the only way back. They're also the single biggest D7 retention lever for any community product.
needs · notifications table · ActionMailer · digest job

Recent vs Hot filter

half day

Segmented control on the feed. Hot = sort by helpful_count desc within last 7 days.

Why now. Cheap, big perceived-quality win. Already mocked in the UI kit.
needs · query param on GET /posts

Three-dot menu → "Report" → reason picker (harm, spam, identifying info). 3 flags auto-hides pending review.

Why now. Anonymous boards will get dumped on. You don't need a real mod team yet — you do need a kill switch.

Open questions for the founder

From the brief — these need a decision before anything in "Next" gets built.

Q1

How do you incentivize the first 100 users to keep coming back?

The reputation system + notifications are the lever. But neither matters without seed content. Possibilities: (a) launch with the founder + close friends seeding 50 honest problems, (b) partner with one campus club, (c) "anonymous AMA" launches with verified people.

My recommendation: Pick one campus. Seed it. Don't open public registration until that one campus retains.
Q2

How does anonymity hold in a small friend group?

You named this in the brief: in a 3-person group, the OP is obvious. The audiences.min_audience guardrail (default 5) refuses to publish a friend-anon post if the group is too small. There's no UX way around this — the math leaks the identity.

My recommendation: Hard floor of 5 friends. Below that, the friend-anon toggle is disabled with copy: "Add more friends — fewer than 5 isn't anonymous enough."
Q3

How do you stop wrong advice from polluting AI training data?

You raised this directly. The helpful-mark system is the answer — only comments with ≥ N marks from distinct users feed the training set. But "helpful" ≠ "correct." A confidently wrong reply can still get hearts.

My recommendation: Don't train an AI until you have a separate annotation layer (clinician / counselor review on a sample). Until then, surface the top-3 community replies; don't auto-generate.
Q4

What does monetization look like that isn't ads?

From the brief. Three plausible paths: premium individual ($3–5/mo, drafts/filters), college site licenses (counseling services pay for aggregate insights), verified-expert AMAs (sponsored — a therapist hosts a thread, the platform takes a cut).

My recommendation: Aim for college site licenses. Highest ARPU, defensible, aligned with the brand. Premium individual is a fallback.
Q5

What's the canonical name? IAP / TAP / ATP / thatsanproblem.com?

The brief lists four. The repo is named itsanproblem. The frontend HTML title is "It's a problem." The design system lands on "It's An Problem" with the intentional grammatical play.

My recommendation: Pick one. "It's An Problem" (full name) + IAP (shorthand) + thatsanproblem.com (URL). Drop TAP/ATP — three names is one too many.

Risks worth naming be honest about these

From the founder brief + things I noticed reading the code.

Anonymity is a DB layout, not a feature.

Today, posts.user_id is a non-null FK. The "anonymous" label is purely cosmetic — set in Post#as_json.

Fix: Move authorship off posts entirely. Use anon_handles for display, and an opt-in post_authors ledger if you need recovery. Don't trust serialization alone.

Cold start = no posts = no one returns.

Anonymous boards die without inertia. A user who visits twice and sees the same three posts is gone.

Fix: Seed manually with 50+ real problems before public launch. The single-campus wedge solves this.

Wrong advice will hurt people.

The brief acknowledges this. Mental-health adjacent tags (suicidal ideation, self-harm, abuse) need a different track than "my roommate smells."

Fix: Crisis-keyword detector → static safety banner ("If you're in crisis, please call …"). Optionally block public replies on flagged posts pending human review.

The friend-anon use case can be weaponized.

"Anonymous feedback to friends" can become anonymous bullying. The 5-friend floor helps but doesn't fix it.

Fix: Friend-anon replies must be marked "helpful" by the recipient to remain visible. Negative-only feedback can be muted by recipient.

Helpful marks ≠ helpful comments.

Reddit-style upvoting rewards confidence and snark over correctness.

Fix: Only the OP's mark counts toward commenter reputation. Other users' marks count for sorting only. Means: rep is "the person you helped said you helped," not "the crowd liked you."

The codebase has zero tests.

spec/ exists but is empty. No CI. Adding features to this base is going to be load-bearing on you not breaking things.

Fix: Before P0 work, add RSpec request specs for the 5 existing endpoints + GitHub Actions CI. Otherwise every new table is a regression risk.
